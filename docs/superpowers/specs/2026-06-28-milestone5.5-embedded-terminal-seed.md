# Milestone 5.5 · Embedded terminal (libghostty) — design seed

> **Status: design seed, not a finalized spec.** This captures the research + recommended
> architecture produced during the Milestone 5 design workflow so a fresh session can start
> M5.5 without re-deriving it. Begin M5.5 by brainstorming/refining this into a spec → plan →
> implementation (TDD), the same flow used for 5A/5B/5C.
>
> **Naming:** this is **Milestone 5.5** (slotted between M5 and the already-planned M6), per
> the user's decision — it is *not* the roadmap's M6.

## Goal

Add a real in-app interactive terminal so the lifecycle affordances that 5B/5C ship as
*interims* become real:

- **Interactive attach** — `container start -ai <id>` (5B's attach is currently read-only
  `logs --follow`).
- **Exec shell** — `container exec -it <id> sh` (5B/5C surface a **disabled** "Open Shell"
  gated on the injected `terminalAvailable` seam, default `false`).
- **Retry in terminal** — `RecoveryAction.retryInTerminal` (currently a clipboard-copy
  interim) runs the command in the embedded terminal.

Shipping M5.5 means: implement the engine behind a port, then **flip `terminalAvailable`
→ true** and route those three affordances to the terminal — no Domain/UI redesign needed.

## Research: libghostty (verified during the M5 design workflow)

- **Real & MIT-licensed**, production-grade runtime: the full embedding `libghostty`
  (surface creation, GPU/Metal rendering, input, internal PTY) is what Ghostty.app itself
  ships, via a `GhosttyKit.xcframework`. **But** the public embedding **C API is pre-1.0 and
  explicitly unstable** — no tagged `libghostty` release (as of mid-2026), no official
  embedding docs, maintainers warn of breaking changes, and the static lib is large (~100 MB+,
  built with the Zig toolchain). Summary: stable engine, unstable/unversioned public surface.
- **Embedding mechanics (same pattern Ghostty.app uses):** link `GhosttyKit.xcframework`
  (`ghostty.h` + `module.modulemap`, imported as a Clang module). Lifecycle:
  `ghostty_init` → `ghostty_app_new(&runtime_config, config)` → `ghostty_surface_new(app,
  &surface_config)` per pane. Host the surface in an `NSView` and expose it to SwiftUI with
  `NSViewRepresentable`; libghostty renders into the layer with its **own Metal renderer**
  (you write no rendering). C→Swift events come through the `ghostty_runtime_config_s`
  callbacks (`wakeup_cb`, `action_cb`, clipboard cbs, `close_surface_cb` fires when the child
  exits); a `userdata void*` carries a pointer to your Swift wrapper. Input via
  `ghostty_surface_key`/`_text`/`_mouse_*`; geometry via `ghostty_surface_set_size` +
  `scale_factor`.
- **Spawning the container command:** `ghostty_surface_config_s` has `command`,
  `working_directory`, `env_vars`, etc. Set `command` to `container exec -it <id> sh` or
  `container start -ai <id>` and **libghostty spawns it in a PTY internally**, renders output,
  and fires `close_surface_cb` on exit. (Confirm the exact `command` semantics — shell-string
  vs argv, quoting — for the pinned build; treat leniently, the config shape can change.)
- **Build/linking:** `libghostty` is built with Zig (`zig build` → `GhosttyKit.xcframework`).
  **Do NOT add Zig to Capsule's normal build.** Two practical routes: (1) **pin a prebuilt
  `GhosttyKit.xcframework`** as a SwiftPM `binaryTarget` (build once / in CI, host or vendor
  it; XcodeGen adds the framework to the **app target only**); or (2) consume a third-party
  prebuilt SwiftPM package (`libghostty-spm`, or `Termini` which bundles a SwiftUI surface +
  local PTY). Pin an exact version either way.
- **Licensing/notarization:** MIT throughout; Ghostty.app ships notarized with Hardened
  Runtime + Metal, so an embedded `GhosttyKit` is no notarization blocker.

### Fallback engine — SwiftTerm

`migueldeicaza/SwiftTerm` is a mature, **MIT, actively-maintained, pure-Swift** NSView
terminal with a **stable API**. `LocalProcessTerminalView.startProcess(executable:"container",
args:["exec","-it",id,"sh"])` spawns the command in a PTY today — no C interop, no Zig, trivial
SwiftPM/XcodeGen integration. It gives you a working vertical slice immediately.

## Recommended architecture

- **New adapter module `CapsuleTerminal`** at the UI/adapter layer — all engine code
  (C-interop / NSView / Metal for libghostty, or SwiftTerm's AppKit view) lives here, wired
  **only by the composition root**. It must **never** be imported by `CapsuleDomain`
  (Domain forbids UI and `Foundation.Process`; the PTY/subprocess spawning must not leak there).
- **A terminal-session port** (a protocol/closure seam injected by the composition root,
  mirroring how `ScopeStore` / `ErrorNormalizer` / the `terminalAvailable` and
  `copyCommand` seams are injected today) so `CapsuleUI`/`CapsuleDomain` never depend on a
  specific engine. The existing `terminalAvailable: () -> Bool` seam (currently `{ false }` in
  `AppEnvironment.live()`) becomes the flag the engine flips on.
- **Engine choice:** build the **port first**; recommend implementing **SwiftTerm first** for
  a working, low-risk vertical slice (a real shell in-app), then prototype **libghostty behind
  the same port** via a pinned `GhosttyKit.xcframework`. Because the port abstracts the engine,
  you can ship SwiftTerm and swap to libghostty later **with no Domain/UI change**. If
  libghostty's API churn / build burden proves painful, SwiftTerm carries the milestone.

## Integration points already prepared by 5A/5B/5C

- **Activity pane reserved terminal slot** — `Sources/CapsuleUI/ActivityPaneView.swift` (the
  pane already has Logs/Tasks/Progress tabs + a reserved slot; `AttachConsoleView` shows when
  attached). Render the terminal surface here.
- **`terminalAvailable` seam** — injected in `AppEnvironment.live()` as `{ false }`; flip to
  `{ true }` (or wire to engine availability) and the disabled "Open Shell" affordances enable.
- **Affordances to route to the port:** `StartAttachSheet` (exec-shell hint), `AttachConsoleView`
  (disabled "Open Shell"), `LifecycleNoticeView` + `ContainerLifecycleModel.copyToTerminal`
  (retry-in-terminal), and 5B's read-only attach (`AttachSession.isReadOnly` — the flag M5.5
  flips for interactive attach).

## Suggested decomposition (refine when you start)

1. **Terminal-session port + seam + composition wiring** behind a feature flag (no engine yet);
   unit-test the seam/routing with a fake terminal session.
2. **Engine adapter** in `CapsuleTerminal` — SwiftTerm first (SwiftPM dep + `NSViewRepresentable`
   surface + PTY via `LocalProcessTerminalView`), rendered in the Activity terminal slot.
3. **Wire the affordances** — exec-shell (`exec -it`), interactive attach (`start -ai`), and
   retry-in-terminal to the port; flip `terminalAvailable`.
4. **(Optional / later)** libghostty behind the port via a pinned `GhosttyKit.xcframework`.

## Risks

- libghostty embedding C API is pre-1.0/unstable, unversioned, ~100 MB Zig build → mitigated by
  the port abstraction + a pinned xcframework + the SwiftTerm fallback.
- PTY/subprocess spawning must stay out of `CapsuleDomain` (arch guard) — keep it in
  `CapsuleTerminal` and behind the port.
- macOS-26 / Apple-silicon / Hardened-Runtime / notarized envelope — all compatible per the
  research, but validate the xcframework signs/notarizes cleanly in `make app`.
