# Milestone 5.5 · Embedded terminal (SwiftTerm) — design

> **Status: approved design (spec).** Refined from the design seed
> (`2026-06-28-milestone5.5-embedded-terminal-seed.md`) through the brainstorming flow.
> Next step: `writing-plans` → TDD implementation.
>
> **Naming:** this is **Milestone 5.5** (slotted between M5 and the already-planned M6), per
> the user's decision — it is *not* the roadmap's M6. Source comments/strings that still say
> "M6" / "the embedded terminal update" refer to this work.

## Goal

Add a real in-app interactive terminal so the lifecycle affordances that 5B/5C shipped as
*interims* become real:

- **Exec shell** — `container exec -it <id> sh` (the disabled "Open Shell" affordance).
- **Interactive attach** — `container start -ai <id>` (today attach is read-only `logs --follow`).
- **Retry in terminal** — `RecoveryAction.retryInTerminal` runs the recovery argv in the
  embedded terminal (today it copies the command to the clipboard).

Shipping M5.5 means: implement an engine behind a port, **flip the `terminalAvailable` seam
`{ false }` → `{ true }`**, and route those three affordances to the terminal — no
Domain redesign.

## Decisions (locked during brainstorming, 2026-06-28)

1. **Engine: SwiftTerm only.** Ship a working SwiftTerm vertical slice behind the port.
   libghostty (pinned `GhosttyKit.xcframework`) is deferred to a later milestone; because the
   port abstracts the engine, it can be swapped in with no Domain/UI change.
2. **Placement: resizable/taller Activity pane.** The terminal reuses the Activity pane's
   reserved slot but grows to a usable height and is drag-resizable when active. (The pane is
   a fixed 160 pt today — ~10 lines, too small for interactive use.)
3. **Attach model: read-only attach stays separate.** The honest read-only `logs --follow`
   console (`AttachConsoleView`) is unchanged. The new interactive terminal is one facility
   with three triggers (exec shell / interactive attach / retry-in-terminal).

## Architecture & dependency direction

A new **`CapsuleTerminal`** adapter module holds *all* engine code (the SwiftTerm dependency,
the `NSViewRepresentable`, PTY/subprocess spawning). Layering and ownership:

```
CapsuleDomain   → TerminalRequest                 (pure value type: argv + container id + kind)
                  openTerminal closure seam        (intent only; never a view, never Process)
CapsuleUI       → TerminalSurfaceProviding         (port protocol; returns an erased view)
                  ShellState.terminalSession        (active-session UI state)
                  ActivityPaneView                  (renders the surface in the reserved slot)
                  StubTerminalSurfaceProvider       (previews/tests)
CapsuleTerminal → SwiftTermSurfaceProvider          (conforms to the port; engine lives here)
                  imports CapsuleUI + CapsuleDomain + SwiftTerm
CapsuleApp      → wires the provider + seams; flips terminalAvailable → true
                  the ONLY module that imports CapsuleTerminal
```

The port returns an **erased SwiftUI view** (`AnyView`), so `CapsuleUI` renders a terminal
*without importing the engine*. `CapsuleTerminal` depends on `CapsuleUI` (allowed — it is an
adapter that produces UI surfaces) and on `CapsuleDomain` (for `TerminalRequest`).
`CapsuleDomain` stays free of UI and `Foundation.Process`.

### Why this shape

It mirrors the patterns already in the codebase:

- **Closure seams injected by the composition root** — `terminalAvailable: @MainActor () -> Bool`,
  `copyCommand: @MainActor ([String]) -> Void`, `onActivity: @MainActor (String) -> Void` are
  all closures captured in `AppEnvironment.live()`. The `openTerminal` intent seam is the same
  pattern: a closure the composition root wires to `ShellState.openTerminal(_:)`.
- **Protocol ports for swappable adapters** — `ContainerBackend` (CapsuleBackend) and
  `ScopeStore` (CapsuleDomain) are `Sendable` protocols with concrete impls in adapter/comp
  modules. `TerminalSurfaceProviding` follows suit; `SwiftTermSurfaceProvider` is its
  `CLIContainerBackend`/`UserDefaultsScopeStore` analogue, and `StubTerminalSurfaceProvider`
  its `MockBackend`/`InMemoryScopeStore` analogue.

## The port & seams

### `TerminalRequest` — CapsuleDomain (pure value type)

```swift
public struct TerminalRequest: Sendable, Equatable, Identifiable {
    public enum Kind: Sendable, Equatable { case execShell, interactiveAttach, retry }
    public let id: UUID            // distinct per launch (drives view identity / restart)
    public let containerID: String?
    public let title: String       // e.g. "Shell · c1"
    public let argv: [String]      // e.g. ["container", "exec", "-it", "c1", "sh"]
    public let kind: Kind
}
```

`argv` is the full command including the `container` executable, matching the existing
`RecoveryAction.retryInTerminal(command:)` convention (`["container", "kill", id]`).

### `TerminalSurfaceProviding` — CapsuleUI (port protocol)

```swift
@MainActor public protocol TerminalSurfaceProviding {
    /// Builds an embeddable terminal surface that spawns `request.argv` in a PTY.
    /// `onExit` fires with the child's termination status when the session ends.
    func makeSurface(for request: TerminalRequest,
                     onExit: @escaping (TerminalExitStatus) -> Void) -> AnyView
}
```

`TerminalExitStatus` (CapsuleUI, alongside the port — it drives the UI banner) carries the
exit code / signal for the "Session ended" banner. A `StubTerminalSurfaceProvider` returns a
placeholder view for previews and any UI-level test.

### Intent + availability seams (composition-wired closures)

- **`openTerminal`** — new `@MainActor (TerminalRequest) -> Void` seam injected into the domain
  model(s). The composition root wires it to `ShellState.openTerminal(_:)`. The domain builds
  the `TerminalRequest` (argv, kind, title) and calls the seam; it never sees a view.
- **`terminalAvailable`** — existing `@MainActor () -> Bool` seam; flips `{ false }` → `{ true }`.
- **`copyToTerminal`/`copyCommand`** — retained as the **fallback** path. When
  `terminalAvailable` is false (e.g. a future headless/preview config), retry-in-terminal still
  copies to the clipboard. This keeps existing behavior and tests meaningful.

## UX — resizable Activity pane

`ActivityPaneView` today is a fixed-height (160 pt) `VStack` with a segmented picker
(Logs / Tasks / Progress) and a `Group { if attachSession { AttachConsoleView } else { content } }`
reserved slot.

Changes:

- **Terminal tab** — a "Terminal" segment appears in the picker **only while a terminal session
  is active**, and is auto-selected when a session opens (`ShellState.terminalSession != nil`).
- **Height** — when the Terminal tab is active, the pane uses a larger default height
  (~320 pt) that is **drag-resizable** (persisted via `@AppStorage`/`ShellState`). Other tabs
  keep 160 pt. A drag handle on the top divider sets the height.
- **Single slot** — one terminal session at a time. Opening a new command replaces the current
  session (terminating the prior PTY). This keeps the single-slot model simple; multiplexing is
  out of scope.
- **Exit handling** — when the child process exits, `onExit` fires; the pane shows an inline
  **"Session ended (status)"** banner with **Restart** (re-launch the same `TerminalRequest`
  with a fresh `id`) and **Close** (clear `terminalSession`, hide the tab).
- Read-only `AttachConsoleView` (`logs --follow`) is unchanged and continues to overlay the
  slot independently when an `AttachSession` is active.

## Affordance routing

| Affordance | Today | M5.5 |
|---|---|---|
| `AttachConsoleView` "Open Shell" | `Button(action: {})`, `.disabled(!terminalAvailable)` | enabled → `openTerminal(.execShell)` = `container exec -it <id> sh` |
| Interactive attach | read-only `logs --follow` console | "Attach interactively" action → `openTerminal(.interactiveAttach)` = `container start -ai <id>` |
| `RecoveryAction.retryInTerminal(command:)` | `lifecycleModel.copyToTerminal` (clipboard) | `openTerminal(.retry)` runs the recovery argv; clipboard remains the fallback when `terminalAvailable` is false |
| `StartAttachSheet` / `LifecycleNoticeView` hint copy | "An interactive shell arrives with the embedded terminal update." | copy updated now that the shell is live |

`AppEnvironment.makeActions` currently has a catch-all that logs `retryInTerminal` as "not
available yet"; that branch routes to `openTerminal` when available.

## Module / build wiring

- **`Package.swift`** — add product `.library(name: "CapsuleTerminal", targets: ["CapsuleTerminal"])`
  and target `.target(name: "CapsuleTerminal", dependencies: ["CapsuleUI", "CapsuleDomain", .product(name: "SwiftTerm", package: "SwiftTerm")])`.
  Add the SwiftTerm SwiftPM dependency at the package `dependencies:` level (the project's first
  external dependency — pin an exact tagged version, e.g. `.package(url: "https://github.com/migueldeicaza/SwiftTerm.git", exact: "<pinned-tag>")`;
  the exact tag is chosen at plan time against the then-current release). Add `CapsuleTerminal`
  to `CapsuleApp`'s dependencies.
- **`App/project.yml`** (XcodeGen) — the app target already consumes the `CapsuleApp` product;
  ensure the SwiftTerm package and `CapsuleTerminal` resolve for the app target. Since
  `CapsuleApp` depends on `CapsuleTerminal` which depends on SwiftTerm, the app target picks it
  up transitively; verify `make xcodeproj` + `make app` resolve and link it.
- **Arch guard** (`Scripts/check-architecture.sh` + `Tests/CapsuleUnitTests/ArchitectureGuardTests.swift`)
  — add `forbid_import CapsuleDomain CapsuleTerminal` and a matching
  `testDomainDoesNotImportCapsuleTerminal()`. Existing rules (Domain ↛ UI / CLIBackend /
  `Process(`; UI ↛ Backend / CLIBackend) stay. `CapsuleTerminal` is *permitted* `Process`/PTY.
  Add `CapsuleTerminal` to the `testGuardActuallyFoundSources()` non-empty check.

## Testing

- **Domain unit tests — the "fake terminal session".** Inject a recording `openTerminal` closure
  (or a small fake) into the lifecycle model. Assert each affordance produces the correct
  `TerminalRequest`:
  - exec-shell → `argv == ["container","exec","-it",id,"sh"]`, `kind == .execShell`
  - interactive attach → `argv == ["container","start","-ai",id]`, `kind == .interactiveAttach`
  - retry → `argv == <recovery command>`, `kind == .retry`
  - gating: `terminalAvailable == true` routes retry to `openTerminal`; `false` falls back to
    `copyCommand` (clipboard).
- **Arch-guard test** — `CapsuleDomain` does not import `CapsuleTerminal`; Domain ↛ `Process`/UI
  still holds.
- **`StubTerminalSurfaceProvider`** — used by previews and any UI test; returns a placeholder.
- **CI** — `make ci` (build, lint, arch, headers, test) green.
- **Manual smoke** — `make app`, open a shell against a running container, confirm input/output,
  resize, exit banner + restart.

## Decomposition (TDD order)

1. **Port + seam + wiring (no engine).** `TerminalRequest`, `TerminalSurfaceProviding`,
   `StubTerminalSurfaceProvider`, `ShellState.terminalSession` + `openTerminal(_:)`, the
   `openTerminal` domain seam, arch-guard rule + test. Domain tests with the fake. CI green.
2. **`CapsuleTerminal` engine.** SwiftPM SwiftTerm dep; `SwiftTermSurfaceProvider`
   (`NSViewRepresentable` over `LocalProcessTerminalView`, `startProcess`, `processTerminated`
   → `onExit`). Builds and links.
3. **Activity-pane integration.** Terminal tab, resizable height, exit/restart banner; render
   the injected surface.
4. **Route affordances + flip the flag.** Wire exec-shell / interactive-attach / retry to
   `openTerminal`; refresh interim copy; flip `terminalAvailable → true`; wire
   `SwiftTermSurfaceProvider` in `AppEnvironment`.

## Risks & mitigations

- **SwiftTerm API specifics** (argv vs shell-string for `startProcess`, the
  `LocalProcessTerminalViewDelegate.processTerminated` signature, scrollback, dark-mode theming)
  — verified at TDD time against the pinned version; isolated inside `CapsuleTerminal`.
- **First external SwiftPM dependency.** Must resolve in *both* `swift build` and the XcodeGen
  app target; validate `make app` links and the dep doesn't break notarization (SwiftTerm is
  MIT, pure-Swift — no Hardened-Runtime/Metal concerns like libghostty would add).
- **Single-slot replace semantics** could surprise a user with a live shell open; acceptable for
  M5.5, revisitable if multiplexing is wanted later.
- **Arch leak.** PTY/subprocess spawning must not drift into `CapsuleDomain`; the guard rule +
  test enforce it.
